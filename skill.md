# IT-Helpdesk Project Skills & Guidelines

> [!IMPORTANT]
> ไฟล์นี้คือคู่มือหลักสำหรับ AI Assistant (Antigravity) ในการพัฒนาและดูแลรักษาโปรเจค IT-Helpdesk
> โปรดอ่านและปฏิบัติตามกฎในนี้อย่างเคร่งครัดทุกครั้งที่ได้รับมอบหมายงาน

## 1. Project Overview & Tech Stack
- **Project Name:** IT Service Helpdesk System
- **Backend:** PHP 8.x (running in Docker)
- **Database:** MySQL 8.0 (Container: `helpdesk-db`, Port: 3306)
- **Frontend:** HTML5, Vanilla CSS, Javascript (เน้นความเร็วและไม่ต้องโหลด Library ภายนอกมากเกินไป)
- **Integrations:** Telegram Bot (Primary notification channel)
- **Environment:** Dockerized environment (Web: 9090, DB: 3308, PMA: 9091)

## 2. Coding Standards (PHP)
- **Security First:**
    - ใช้ **PDO with Prepared Statements** 100% เพื่อป้องกัน SQL Injection
    - ทำการ Sanitize input ทุกครั้งที่รับจาก $_POST หรือ $_GET
- **Connection Style:**
    - อ้างอิงค่าจาก `config.php` เสมอ
    - ใช้ `DB_HOST`, `DB_NAME`, `DB_USER`, `DB_PASS`
- **Error Handling:**
    - ใช้ `try-catch` block ในส่วนที่ติดต่อกับ Database
    - ห้ามแสดง Raw Error ให้ User เห็น (ใช้ User-friendly message แทน)

## 3. UI/UX & Design Philosophy
- **Aesthetics:** "Modern, Clean, and Professional"
- **Color Palette:**
    - Primary: Blue (#2563eb)
    - Background: Light Gray (#f8fafc) หรือ Dark Mode (#1e293b)
- **Interactivity:**
    - ใช้ CSS Transitions สำหรับ Hover effects และ Modal
    - ใช้ JavaScript (Fetch API) สำหรับการอัปเดตข้อมูลแบบไม่รีเฟรชหน้า (ถ้าทำได้)
- **Responsive:** ทุกหน้าจอต้องทำงานได้ดีทั้งบน Desktop และ Mobile

## 4. Database Workflow
- **Schema Changes:**
    - หากมีการเพิ่ม Column หรือ Table ให้สร้างไฟล์ SQL ไว้ใน `docker/mysql/`
    - หรือเขียนไฟล์ PHP สั้นๆ (เช่น `add_column.php`) สำหรับรัน One-time migration
- **Naming Convention:**
    - Table: snake_case (เช่น `it_tickets`, `user_accounts`)
    - Column: snake_case
- **Query & Test Optimization:**
    - หลีกเลี่ยงการดึงข้อมูลขนาดใหญ่หรือรัน Query ครอบคลุมหลายปีซึ่งอาจทำให้ MySQL timeout
    - การทดสอบเขียนและรัน SQL Query หรือทำ Automated tests ทั้งหมด **ต้องทำบนข้อมูลตัวอย่าง (Data Sample) ขนาดเล็กที่มีความสำคัญ/เป็นตัวแทนที่ดีเท่านั้น (เช่น ช่วงเวลาไม่เกิน 1 เดือน)** เพื่อลดปริมาณการรันและป้องกันปัญหาระบบค้างหรือโควต้า AI หมดเร็วเกินไป

## 5. Notification & Logs
- **Telegram Notify:**
    - เมื่อมี Ticket ใหม่ หรือสถานะเปลี่ยน ให้ส่งแจ้งเตือนผ่าน Telegram Bot เสมอ
    - ใช้ฟังก์ชันใน `config.php` หรือฟังก์ชันส่วนกลางที่เตรียมไว้
- **Logging:**
    - บันทึกการเปลี่ยนแปลงสำคัญลงใน `TROUBLESHOOTING_LOG.md` หรือ `CONFIGURATION_LOG.md` หาก AI เป็นผู้ดำเนินการ

## 6. How to work with me (AI Assistant)
- **Before starting:** ตรวจสอบไฟล์ `docker-compose.yml` เพื่อดูความเชื่อมโยงของ Service
- **When writing code:** ตรวจสอบโค้ดใน `./src` ก่อนเสมอ เพื่อให้เขียนได้สอดคล้องกับโครงสร้างเดิม
- **After finish:** ทดสอบการรันผ่าน Docker และสรุปสิ่งที่ทำลงใน `README.md` หรือ Log ไฟล์

## 7. Karpathy Behavioral Guidelines (แนวทางปฏิบัติตามแนวคิด Karpathy)
เพื่อลดความผิดพลาดทั่วไปของ AI ในการเขียนโค้ด (เน้นความรอบคอบและความเรียบง่าย):
- **Think Before Coding (คิดก่อนเริ่มเขียนโค้ด):**
    - ห้ามเดาหรือคิดแทนเอง: ระบุสมมติฐานให้ชัดเจน หากไม่แน่ใจหรือสับสนจุดใด ให้หยุดถามผู้ใช้ทันทีก่อนเขียนโค้ด
    - เสนอทางเลือกที่ดีกว่า/เรียบง่ายกว่า: หากการตีความมีความกำกวมหรือมีหลายทางเลือก ให้แสดงทางเลือกเพื่อรับความเห็นชอบก่อน อย่าตัดสินใจเงียบๆ
- **Simplicity First (เน้นความเรียบง่ายเป็นอันดับแรก):**
    - เขียนโค้ดให้น้อยที่สุดที่แก้ปัญหาได้ โดยไม่มีการคาดเดาหรือสร้าง abstraction เผื่อฟีเจอร์ในอนาคตที่ไม่ได้ร้องขอ
    - หากเขียนโค้ด 200 บรรทัด แต่สามารถจัดกลุ่ม/เขียนให้เหลือ 50 บรรทัดได้ ให้เขียนใหม่เพื่อให้สั้นกระชับและเข้าใจง่ายที่สุด
- **Surgical Changes (การแก้ไขโค้ดเฉพาะจุดแบบแม่นยำ):**
    - แก้ไขเฉพาะบรรทัดที่จำเป็นต้องแก้และเกี่ยวข้องโดยตรงกับคำร้องขอเท่านั้น ห้ามเข้าไปจัดฟอร์แมตหรือปรับปรุงโค้ดส่วนอื่นๆ ที่ทำงานได้ดีอยู่แล้วโดยไม่จำเป็น
    - เมื่อทำการลบหรือเปลี่ยนโค้ดแล้ว หากทำให้ตัวแปร/ฟังก์ชันใดไม่ได้ใช้งาน (Orphans) ให้ทำความสะอาดเฉพาะส่วนที่เกิดจากการแก้ไขของเรา
- **Goal-Driven Execution (การรันงานแบบกำหนดเป้าหมายและตรวจสอบได้):**
    - กำหนดตัวชี้วัดความสำเร็จ (Success Criteria) ให้เป็นรูปธรรมก่อนเริ่มทำ เช่น การเขียนหรือระบุการทดสอบที่ยืนยันว่าฟีเจอร์ทำงานถูกต้อง
    - วางแผนขั้นตอนสั้นๆ (Step-by-step plan) พร้อมวิธีตรวจสอบในแต่ละขั้นตอนก่อนดำเนินการเสมอ

## 8. Caveman Mode (แนวปฏิบัติแบบโหมดมนุษย์ถ้ำ)
เพื่อประหยัดโควต้าโทเค็นและเพิ่มความรวดเร็วในการทำงาน (ลดการใช้โทเค็นประมาณ 75% โดยยังคงความถูกต้องทางเทคนิคครบถ้วน):
- **หลักการสำคัญ:** ตอบคำถามสั้นกระชับเหมือนมนุษย์ถ้ำฉลาด ตัดคำฟุ่มเฟือยทั้งหมดออก เหลือเฉพาะเนื้อหาทางเทคนิคที่จำเป็น
- **กฎเกณฑ์:**
    - ตัดคำนำหน้านาม (a/an/the) คำขยายหรือคำเติมเต็มที่ไม่จำเป็น (just/really/basically) คำสุภาพทักทาย และคำออกตัว (hedging)
    - สามารถใช้ประโยคที่ไม่สมบูรณ์ (Fragments) คำพ้องเสียง/คำสั้น และรักษาศัพท์เทคนิคให้ถูกต้อง ส่วนโค้ดห้ามตัดทอนหรือแก้ไข
    - รูปแบบการตอบ: `[หัวเรื่อง/ปัญหา] [การกระทำ/สิ่งที่ต้องทำ] [เหตุผล]. [ขั้นตอนถัดไป].`
    - ตัวอย่าง:
        - *แบบทั่วไป:* "ระบบ React ของคุณกำลัง re-render เพราะมีการสร้าง object reference ใหม่ในทุกรอบ แนะนำให้ใช้ useMemo ครับ"
        - *แบบมนุษย์ถ้ำ:* "ออบเจกต์สร้างใหม่ทุกเรนเดอร์. Prop ออบเจกต์แบบ inline = อ้างอิงใหม่ = เรนเดอร์ซ้ำ. ครอบด้วย `useMemo`."
- **ข้อยกเว้น (Auto-Clarity):**
    - ให้เปลี่ยนกลับไปใช้การอธิบายแบบปกติเมื่อ: แจ้งเตือนความปลอดภัย ยืนยันการกระทำที่ลบข้อมูล หรือเมื่อลำดับขั้นตอนซับซ้อนเกินกว่าจะใช้รูปประโยคสั้นๆ เพื่อหลีกเลี่ยงความเข้าใจผิด

## 9. RTK (Rust Token Killer) Guidelines
เพื่อป้องกันไม่ให้โควต้าโทเค็นของ AI หมดลงอย่างรวดเร็ว และทำให้การส่งผลลัพธ์ของคำสั่งรวดเร็วยิ่งขึ้น:
- **หลักการทำงาน:** RTK เป็น CLI proxy ที่จะกรองและบีบอัดผลลัพธ์ของคำสั่งในระบบปฏิบัติการ ก่อนที่จะถูกส่งเข้าไปใน Context ของ AI ช่วยลดจำนวน Token ได้ประมาณ 60-90%
- **กติกาบังคับใช้:**
    - เสมอเมื่อใช้คำสั่ง Shell ให้ครอบคำสั่งนั้นๆ ด้วยคำนำหน้า `rtk` (เช่น `rtk git status`, `rtk ls src/`, `rtk grep`, `rtk find`, `rtk docker ps`)
    - ยกเว้นกรณีคำสั่งที่ต้องการผลลัพธ์ดิบจริงๆ หรือคำสั่งที่ RTK ไม่รองรับ ให้ใช้ `rtk proxy <cmd>` หรือรันคำสั่งโดยไม่มี `rtk` นำหน้า
- **คำสั่งที่มีประโยชน์ของ RTK:**
    - `rtk gain` เพื่อดูรายงานการประหยัดโทเค็นสะสม
    - `rtk discover` เพื่อค้นหาคำสั่งที่อาจจะลืมใส่ `rtk` นำหน้า
- **การแสดงรายการข้อมูล:**
    - หากต้องแสดงรายการข้อมูล/ผลลัพธ์ใดๆ ให้ใช้รูปแบบ JSON ที่กระชับ (อิงตามมาตรฐาน `ai-native-cli`) เพื่อลดขนาดของข้อความและข้อความขยะให้น้อยที่สุด
